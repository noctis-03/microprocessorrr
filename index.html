02프로젝트 개요 & 문제 정의

출발점 — 종이 캘린더의 한계

기존 종이 캘린더는 날짜와 일정만을 정적으로 표시할 뿐, 그날의 날씨·미세먼지·온습도 같은 실시간 환경 정보를 전혀 담지 못합니다. 또한 사용자가 직접 매달 종이를 넘기거나 글씨를 써넣어야 하며, 한 번 인쇄된 내용은 갱신이 불가능합니다.

그렇다면 LCD/태블릿으로 대체하면?

  • 정보를 항상 띄워두려면 전력 소모가 커서 배터리로 오래 운용 불가
  • 발광형 화면이라 눈 피로가 누적되고 종이 같은 자연스러운 가독성이 없음
  • 매번 사람이 직접 앱을 켜야 하므로 "고개만 돌리면 항상 보이는 즉시성" 상실

핵심 문제 정의

"종이 캘린더처럼 전력 없이도 항상 정보가 떠 있고 눈이 편안하면서도, 종이 캘린더가 하지 못하는 실시간 날씨·환경 정보 갱신과 자동화를 저전력 제약 안에서 어떻게 동시에 구현할 것인가?"

해결 방향 — 두뇌와 표시의 역할 분리

A 보드 (표시 단말)

종이의 장점은 살리고 한계는 극복하는 e-paper 디스플레이. 쌍안정성 + Deep Sleep으로 1회 충전 후 약 1년 운용.

B 보드 (두뇌/게이트웨이)

저전력 단말을 대신해 인터넷·클라우드·환경 데이터를 책임지는 상시 동작 노드. A·C 두 단말에 정보를 공급.

03관련 기술 현황

E-Paper (전자종이) — 원리

전기영동(Electrophoretic Display) 방식의 화면은 수많은 마이크로캡슐로 채워져 있고, 각 캡슐 안에는 음(−) 전하 흰색 입자와 양(+) 전하 검은색 입자가 떠 있습니다. 전극으로 전기장을 걸어 흰 입자를 올리면 흰색, 검은 입자를 올리면 검은색으로 보입니다.

쌍안정성(Bistability): 한 번 입자를 이동시키면 전원을 꺼도 그 자리에 그대로 머뭅니다. → 화면을 바꿀 때만 전기가 필요하고, 보여주는 동안에는 전기가 거의 들지 않음.

ESP-NOW — Espressif의 커넥션리스 통신

항목특성
구조OSI 5개 상위 계층을 데이터링크 계층 하나로 축약, 라우터 불필요
송신 전력최대 +21 dBm (개활지 약 200m)
지연약 150ms 이내
공존WiFi / BLE 와 공존 가능

저전력윈도우 동기화(window synchronization)로 저전력 동작

외부 API

Open-Meteo

전 세계 기상 예보 제공 오픈소스 무료 API. 비상업 용도에 한해 API 키 불필요 → 가입·인증 없이 즉시 호출 가능.

Gemini API

구글의 LLM(Gemini)을 HTTP 요청으로 호출해 텍스트를 생성받는 클라우드 서비스. 본 시스템은 날씨 수치를 한 줄 생활 조언으로 변환.

05아이디어 창출 — SCAMPER

SSubstitute (대체)
종이 → 전자종이(e-paper)로 대체. 가독성·전원 없는 표시 유지는 살리고 인쇄된 고정 내용은 무선 갱신 가능한 화면으로 전환. → A 보드
CCombine (결합)
날짜에 날씨·미세먼지·온습도·AI 조언을 한 화면에 결합. 이 정보 수집을 위해 인터넷·센서를 담당하는 B 보드가 필요해짐. → A·B
AAdapt (응용)
ESL의 "서버가 다수 저전력 단말 갱신" 구조를 가정용으로 응용. 게이트웨이-단말 분리 구조의 직접적 근거. → B·A
MModify (수정)
표시 단말의 전력 운용을 근본적으로 변경. 평소엔 Deep Sleep, 버튼/정해진 시각에만 깨어남. → A
PPut to another use
캘린더 단일 용도를 넘어 날씨 대시보드·대기질 모니터·AI 비서로 화면 모드 확장. 같은 HW로 다중 화면 제공. → A
EEliminate (제거)
A에서 상시 WiFi·클라우드 호출 제거 → 모두 B로 이관. 라우터 의존 제거 → ESP-NOW 직접 통신 채택. → A·B
RReverse / Rearrange (역발상·재배치)
"사람이 정보를 찾아가는 방식"을 "정보가 사람에게 미리 준비되는 방식"으로 뒤집음. wakeHour에 단말이 스스로 깨어나 그날 정보를 미리 갱신, 날씨 수치를 사람이 해석하던 부담을 AI 한 줄 조언으로 재배치. → B

08사용자 시나리오

  1. 아침 자동 기상 (A ↔ B)

    사용자가 설정한 기상 시각(기본 7시, wakeHour)이 되면 A 보드가 스스로 Deep Sleep에서 깨어남 → B에 날짜·날씨·미세먼지·AI 멘트를 요청 → 화면 갱신 후 다시 잠듦. "사용자가 정보를 찾으러 가는 것이 아니라 정보가 미리 준비되어 있는 것"이 핵심 경험.

  2. 평상시 화면 확인 — 수동 조작 (A ↔ B)

    날씨가 궁금해 버튼을 누르면 A가 즉시 깨어나 최신 데이터를 받아 갱신. UP/DOWN으로 캘린더 → 시간별 날씨 → 대기질+AI 멘트 화면 전환, MENU로 일/월 보기 전환, EXIT로 System Info(배터리·통신 채널·마지막 동기화 시각) 확인. sleepTimeoutSec(기본 10초) 동안 조작이 없으면 캘린더 화면으로 복귀 후 재진입.

  3. 게이트웨이가 꺼져 있을 때 — 통신 실패 폴백

    B의 전원이 꺼져 있어도 A는 마지막 성공 채널부터 1~13 채널을 스캔. 그래도 응답이 없으면 RTC 메모리의 마지막 데이터를 화면에 표시. 우측 상단에 "CACHED" 배지와 마지막 동기화 시각이 함께 표시되어 캐시 상태임을 명확히 알림.

  4. 설정 변경 — 웹 페이지 (B)

    스마트폰으로 B의 WiFi 웹 설정 페이지에 접속해 위치(위도·경도), 기상 시각, 화면 대기시간, AI 멘트 on/off 등을 변경. B는 NVS에 저장하면서 configVersion을 +1. 다음 요청 시 A가 새 설정을 받음. → 단말을 직접 만지지 않고 한곳에서 모든 기기 설정 변경 가능.

  5. 환경 자동 제어 — 선풍기 (C ↔ B)

    C는 두 가지 모드로 동작.
    Default 모드: 초음파로 사람 접근 감지 → 설정된 출력(LED+팬 / 팬만 / LED만)에 따라 동작. 조이스틱으로 팬·조명 방향 조절, 버튼 길게 누르면 가장 가까운 대상으로 자동 정렬.
    AUTO 모드: B의 온도(≥28°C)·조도·PIR 값 기준으로 자동 켜짐. B와 10초 이상 통신 끊기면 페일세이프로 모든 출력 OFF.

09A 보드 — E-Paper 디스플레이

동작 원리

평소 Deep Sleep 상태이며 버튼(EXT0/EXT1 wakeup) 또는 타이머로 깨어남. 깨어나면 채널 자동 스캔으로 B를 찾아 MSG_REQUEST(센서·날씨)와 MSG_AI_REQUEST(AI 멘트)를 전송. B의 응답을 받아 e-paper에 표시하고 RTC 메모리에 캐시한 뒤 다시 잠듦.

렌더링 & 전력 관리

  • 부분 갱신으로 빠르게 그리되, fullRefreshEvery마다 전체 갱신으로 잔상 제거
  • 날씨 코드별 24×24 아이콘(맑음/구름/비/눈/안개/뇌우)과 AI 멘트 워드랩 표시
  • Deep Sleep 진입 직전, 화면 좌측 상단 작은 점으로 슬립 상태 표시 (부분 갱신)

다음 기상 시각(wakeHour) 계산

if (hour < wakeHour)            // 오늘 아직 기상 시각 전
  sec = (wakeHour - hour) * 3600 - minute * 60;
else                            // 이미 지남 → 다음 날 기상
  sec = (24 - hour + wakeHour) * 3600 - minute * 60;

if (sec <= 0) sec = 3600;       // 안전장치: 최소 1시간

예) wakeHour=7, 현재 22:30 → 다음 날 07:00까지 약 8시간 30분 후 기상. NTP 동기화 안 된 상태면 기본 24시간 주기.

10B 보드 — 게이트웨이

핵심 기능

WiFi & AP 모드

저장된 자격증명으로 접속, 실패 시 DualCore-Setup AP를 띄워 웹 설정 수신.

외부 API 호출

Open-Meteo(날씨), Air Quality(대기질), Gemini(생활 조언 1줄)를 호출.

오늘의 기상·대기질을 프롬프트로 넣어 영어 18단어 이내 생활 팁을 하루 1회 생성하고 NVS에 저장 → 재부팅 후에도 유지. ESP-NOW 250바이트 제약 때문에 200바이트씩 최대 4개의 MSG_AI_CHUNK로 분할 전송.

typedef struct {

11C 보드 — 탁상용 선풍기 (추가)

C 보드를 추가한 이유

A 보드를 통해 소프트웨어적 설계와 Deep Sleep의 동작 원리를 이해함. 특히 Deep Sleep 상태에서는 CPU·WiFi가 정지되어 외부로부터 인터럽트(데이터 수신)를 받을 수 없다는 한계를 직접 확인. 즉 A가 잠들어 있는 동안 외부에서 능동적으로 데이터를 밀어 넣을 수 없고, A가 스스로 깨어나 요청하는 방식(pull)만 가능하다는 점을 학습.

학습 동기: 위 내용을 통해 소프트웨어 설계는 이해했으나, 모터·서보·초음파·LED 등 실제 액추에이터를 제어하는 하드웨어적 이해가 부족하다고 느낌. 또한 B 보드가 단순 중계기를 넘어, 자신의 센서(온도·조도·PIR)를 또 다른 노드인 C에게도 동시에 공급하는 진정한 의미의 허브로 기능함을 보여주기 위해 C 보드 추가.

12ESP-NOW 통신 프로토콜

세 보드는 동일한 구조체를 공유하며, 수신 측은 패킷 크기와 msgType으로 종류를 구분. A·C 메시지가 충돌하지 않도록 C 전용 타입은 10·11번으로 분리.

타입값방향설명
MSG_REQUEST0A → B센서 데이터 요청
MSG_SENSOR_DATA1B → A센서·시간·날씨·설정 응답
MSG_AI_REQUEST2A → BAI 멘트 요청
MSG_AI_CHUNK3B → AAI 멘트 분할 전송 (4×200B)
MSG_C_REQUEST10C → BC 전용 센서·설정 요청
MSG_C_DATA11B → C온도·조도·PIR + C 설정 응답

채널 자동 탐색

A·C 모두 고정 채널이 아니라 RTC에 저장된 마지막 성공 채널을 우선 시도하고, 실패하면 1~13 채널을 스캔해 B를 찾음. 성공한 채널은 다시 RTC에 저장. 스캔 실패 시 폴백값은 DEFAULT_CHANNEL = 10.

13설정 동기화 메커니즘 (Version 방식)

B에서 바꾼 설정을 A·C에 안전하게 전파하기 위해 버전 번호(configVersion) 사용.

  • 웹에서 설정 저장 시 B의 configVersion이 +1
  • 설정값은 매 센서 응답에 실려 A·C로 전달
  • A·C는 받은 버전이 현재 적용 버전보다 클 때만 적용
  • 같은 설정이 반복 수신돼도 불필요한 갱신 없이 변경 시점만 정확히 반영
// A·C 보드 공통 수신 콜백
if (rxMsg.config.configVersion > lastCfgVersion) {
  appliedCfg     = rxMsg.config;          // 새 설정 적용
  lastCfgVersion = rxMsg.config.configVersion;
}
왜 버전 방식인가? ESP-NOW는 무선 통신이라 같은 패킷이 여러 번 도착하거나 순서가 바뀔 수 있습니다. 버전 비교로 멱등성(idempotency)을 확보해 설정이 의도치 않게 되돌아가는 현상을 막습니다.

  • WiFi 웹 접속으로 A·B·C 설정 변경 가능

15성능 평가 — 전력 소모 추정

주의: 아래 내용은 성능 실험을 진행할 경우 적용할 수 있는 측정 방법에 대해 논하며, 성능 실험은 예상 전력 소모량 계산으로 대체하였습니다.

실험 구성

A 보드에 리튬 배터리(기준 용량 3000mAh 가정)를 연결하고, USB 전류계 또는 멀티미터를 전원 라인에 직렬로 연결해 세 가지 상태의 전류를 측정.

하루 소비 전력량 계산

사용 패턴 가정: 하루 6회 깨어남 (정기 1회 + 수동 5회), 한 번에 약 15초 활성

활성 소비: 100 mA × (6 × 15초 ÷ 3600) ≈ 2.5 mAh

대기 소비: 0.05 mA × 23.97h ≈ 1.2 mAh

하루 총 소비: 약 3.7 mAh

이론 수명: 3000 mAh ÷ 3.7 ≈ 약 810일 (≈ 2.2년)

자가방전·효율 손실·온도 등을 고려한 안전계수 적용해도 약 1년 이상의 운용이 가능하다는 결론.

16현실적 제한조건 반영

제한조건반영 내용
경제성 무료 API(Open-Meteo) 활용으로 구축 비용 절감. Gemini 무료 사용량 제한으로 하루 1회 문구 생성·저장 후 재사용 → 잘못된 문구나 변동 불일치 가능성 존재.
환경성 전자종이 + Deep Sleep으로 1회 충전 후 장기 운용 → 배터리 교체·폐기 절감. 종이를 매월 교체할 필요가 없어 자원 절감에도 기여.
사회성 날씨·미세먼지 + AI 생활 조언 항시 제공 → 일상 정보 접근성, 건강 관리(마스크 등)에 도움. 별도 앱 조작 없이 표시되어 디지털 취약 계층도 쉽게 이용 가능.
윤리성해당 내용은 프로젝트와 관계 없음.
미학 C 보드(선풍기) 배선이 외부로 드러나 외관 정돈 부족. 3D 모델링·치수 측정·출력 등 맞춤 케이스 제작은 난이도가 높아 외관 마감보다 기능 구현에 우선순위를 둠.
보건/안전 전자종이는 발광형이 아니어서 눈 피로·블루라이트 부담이 적음. C 보드는 통신 두절 시 모든 출력을 끄는 페일세이프를 적용해 모터·팬 오작동 방지.
내구성 A 보드는 Deep Sleep으로 부품 부하가 적어 수명이 길고, 통신 실패 시 캐시 폴백과 채널 자동 스캔으로 불안정한 환경에서도 안정 동작.
관련 표준성 ESP-NOW (Espressif 표준 프로토콜), I²C·SPI, WMO 날씨 코드, US AQI 대기질 기준, REST/JSON API 등 널리 쓰이는 표준 규격 준수.
