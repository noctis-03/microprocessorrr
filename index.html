01프로젝트 개요

책상 위에서 날짜·날씨·대기질·AI 생활 조언을 보여주는 스마트 디스플레이입니다. 역할이 다른 두 개의 ESP32 보드를 ESP-NOW로 연결해, 한쪽은 인터넷 데이터를 모으는 게이트웨이로, 다른 쪽은 배터리로 오래 버티는 저전력 표시 단말로 분리한 것이 핵심 설계입니다.

B 보드게이트웨이 / 상시 동작

WiFi로 인터넷에 연결되어 날씨·대기질 API와 Gemini AI를 호출하고, 센서를 읽으며, 웹 설정 페이지를 제공합니다. OLED로 상태를 실시간 표시합니다.

A 보드디스플레이 / 저전력

평소 Deep Sleep 상태로 대기하다가 정해진 시각이나 버튼 입력에 깨어나, B보드에 데이터를 요청해 e-paper에 그린 뒤 다시 잠듭니다.

왜 두 보드로 나눴나? WiFi와 API 호출은 전력 소모가 크기 때문에, 이 부분을 B보드가 전담하고 A보드는 ESP-NOW 통신과 화면 갱신만 담당하게 하여 A보드의 배터리 수명을 극대화했습니다.

04통신 프로토콜 (ESP-NOW)

두 보드는 동일한 구조체와 메시지 타입을 공유합니다. 수신 측은 패킷 크기로 메시지 종류를 구분합니다.

4.1 메시지 타입

타입값방향설명
MSG_REQUEST0A → B센서 데이터 요청
MSG_SENSOR_DATA1B → A센서·시간·날씨·설정 응답
MSG_AI_REQUEST2A → BAI 멘트 요청
MSG_AI_CHUNK3B → AAI 멘트 분할 전송

4.2 Message 구조체 (메인 데이터)

온습도, 현재 시간, 6개 시간대 날씨 예보, 오늘 최고/최저·강수확률, PM2.5·AQI, 그리고 BoardConfig 설정값까지 한 번에 담아 전달합니다.

4.3 AIChunk — 청크 분할 전송

AI 멘트는 최대 800자(200바이트 × 4)까지 가능하지만 ESP-NOW 한 패킷이 작기 때문에, 멘트를 200바이트 단위로 나눠 순차 전송하고 A보드가 재조립합니다.

typedef struct {
  uint8_t  msgType;       // MSG_AI_CHUNK
  uint8_t  totalChunks;   // 전체 조각 수
  uint8_t  chunkIndex;    // 현재 조각 번호
  uint8_t  dataLen;       // 이 조각의 실제 길이
  uint8_t  commentDay;    // 멘트 생성 날짜(중복 방지)
  char     data[200];     // 조각 데이터
} AIChunk;

4.4 요청–응답 흐름

A보드는 응답이 올 때까지 최대 3회 재시도(SCAN_ATTEMPTS)하며, 센서는 1초·AI는 2초의 타임아웃을 둡니다. 응답이 없으면 RTC 메모리의 캐시 데이터를 사용합니다.

05B 보드 — 게이트웨이 상시 동작

WiFi & AP 모드

저장된 자격증명으로 접속을 시도하고, 실패하거나 설정이 없으면 DualCore-Setup AP를 띄워 웹으로 설정받습니다.

외부 API 연동

Open-Meteo(날씨), Open-Meteo Air Quality(대기질), Google Gemini(생활 조언 1줄)를 호출합니다.

NTP 시간 동기화

한국 시간(KST, GMT+9)으로 시각을 맞추고 A보드에 함께 전달합니다.

웹 설정 페이지

위치·기상 좌표·기상시각·화면 타임아웃·AI on/off 등을 변경하고, 공장 초기화도 가능합니다.

AI 멘트 생성 (Gemini)

오늘의 기상·대기질 데이터를 프롬프트에 넣어 영어 18단어 이내의 생활 팁을 생성합니다. 하루 한 번만 생성하며 NVS에 저장해 재부팅 후에도 유지합니다.

"You are a friendly daily assistant for a desk display in Daejeon, Korea.
 Today's data: high X C, low X C, rain X%, PM2.5 X, US AQI X (grade).
 Write ONE short practical life tip in English, max 18 words..."

06A 보드 — e-paper 디스플레이 저전력

화면 모드 (버튼 네비게이션)

📅 Calendar

일/월 보기 전환 (MENU)

🌤️ Weather

6개 시간대 예보

🌫️ Air/Env

종합 + AI 멘트

UP/DOWN으로 모드 전환, OK로 새로고침, EXIT로 시스템 Info 화면 진입.

e-paper 렌더링 전략

  • 부분 갱신(partial)으로 빠르게 그리되, fullRefreshEvery 횟수마다 한 번씩 전체 갱신으로 잔상을 제거합니다.
  • 날씨 코드별 24×24 비트맵 아이콘(맑음/구름/비/눈/안개/뇌우)을 표시하고, AI 멘트는 워드랩으로 줄바꿈합니다.

Deep Sleep & 전력 관리

화면을 그린 뒤 일정 시간(sleepTimeoutSec) 입력이 없으면 캘린더로 복귀 후 잠듭니다. 다음 기상은 설정된 wakeHour까지의 시간을 계산해 타이머로, 또는 버튼으로 깨어납니다.

RTC 메모리 활용: RTC_DATA_ATTR 변수로 시간·날씨·AI 멘트·설정을 Deep Sleep 동안 유지하므로, 통신이 실패해도 마지막 데이터를 그대로 표시할 수 있습니다.

07설정 동기화 메커니즘 핵심

B보드에서 바꾼 설정을 A보드에 안전하게 전파하기 위해 버전 번호(configVersion) 방식을 사용합니다.

  • 웹에서 설정을 저장하면 B보드의 configVersion이 1 증가합니다.
  • 설정값은 매 센서 응답(Message)에 실려 A보드로 전달됩니다.
  • A보드는 받은 버전이 현재 적용된 버전보다 클 때만 새 설정을 적용합니다.
  • 덕분에 같은 설정이 반복 수신돼도 불필요한 갱신이 없고, 변경 시점만 정확히 반영됩니다.
// A보드 수신 콜백
if (rxMsg.config.configVersion > lastCfgVersion) {
  appliedCfg     = rxMsg.config;          // 새 설정 적용
  lastCfgVersion = rxMsg.config.configVersion;
  savedChannel   = appliedCfg.espnowChannel;
}

→ "A보드는 다음에 깨어날 때 새 설정을 반영합니다"라는 동작이 여기서 구현됩니다.

08동작 시나리오

  1. A보드가 타이머/버튼으로 Deep Sleep에서 기상하고 ESP-NOW를 초기화합니다.
  2. B보드에 센서 데이터를 요청 (온습도·시간·날씨·대기질·설정 수신).
  3. Air/Env 모드이고 AI가 켜져 있으면 AI 멘트를 요청해 청크로 수신·재조립합니다.
  4. 현재 모드에 맞는 화면을 e-paper에 렌더링합니다.
  5. 타임아웃까지 입력이 없으면 캘린더로 복귀 후 Deep Sleep 재진입.
통신 실패 시: 재시도 후에도 응답이 없으면 RTC 메모리의 마지막 데이터(캐시)로 화면을 그려 항상 무언가는 표시되도록 보장합니다.

09빌드 & 설정

  • 라이브러리: WiFi, esp_now, WebServer, Preferences, Adafruit_SSD1306, DHT, ArduinoJson, GxEPD2
  • MAC 주소 매칭: 각 보드 코드의 peerMAC_A / peerMAC_B를 상대 보드 실제 MAC으로 설정
  • ESP-NOW 채널 일치: 두 보드 모두 채널 10 (FIXED_CHANNEL)
  • Gemini API 키: GEMINI_API_KEY에 본인 키 입력 (코드에 직접 노출 주의)

10한계 & 개선점

  • API 키 하드코딩: 소스에 키가 노출되므로 NVS 저장이나 별도 설정 분리가 바람직합니다.
  • AI 멘트 갱신: 하루 1회 생성 — 날씨 급변 시 반영이 늦을 수 있습니다.
  • ESP-NOW 비암호화: 현재 encrypt = false 상태로, 필요 시 암호화 적용 가능.
  • 고정 위치(대전): 좌표는 변경 가능하나 AI 프롬프트 도시명은 고정되어 있습니다.
