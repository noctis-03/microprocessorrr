01프로젝트 개요

책상 위에서 날짜·날씨·대기질·AI 생활 조언을 보여주는 스마트 디스플레이입니다. 역할이 다른 두 ESP32 보드를 ESP-NOW로 연결해, 한쪽은 인터넷 데이터를 모으는 게이트웨이로, 다른 쪽은 배터리로 오래 버티는 저전력 표시 단말로 분리한 것이 핵심 설계입니다.

B 보드게이트웨이 / 상시 동작

WiFi로 날씨·대기질 API와 Gemini AI를 호출하고, 센서를 읽으며, 웹 설정 페이지를 제공합니다. OLED로 상태를 표시합니다.

A 보드디스플레이 / 저전력

평소 Deep Sleep으로 대기하다 정해진 시각이나 버튼에 깨어나, B보드에 데이터를 요청해 e-paper에 그린 뒤 다시 잠듭니다.

왜 두 보드로? WiFi·API 호출은 전력 소모가 크므로 B보드가 전담하고, A보드는 통신과 화면 갱신만 맡겨 배터리 수명을 극대화했습니다.

04통신 프로토콜 (ESP-NOW)

두 보드는 동일한 구조체와 메시지 타입을 공유하며, 수신 측은 패킷 크기로 종류를 구분합니다.

타입값방향설명
MSG_REQUEST0A → B센서 데이터 요청
MSG_SENSOR_DATA1B → A센서·시간·날씨·설정 응답
MSG_AI_REQUEST2A → BAI 멘트 요청
MSG_AI_CHUNK3B → AAI 멘트 분할 전송

AIChunk — 청크 분할 전송

AI 멘트는 ESP-NOW 패킷 크기 제한 때문에 200바이트 단위(최대 4조각)로 나눠 전송하고, A보드가 인덱스를 보고 재조립합니다.

typedef struct {
  uint8_t  msgType;       // MSG_AI_CHUNK
  uint8_t  totalChunks;   // 전체 조각 수
  uint8_t  chunkIndex;    // 현재 조각 번호
  uint8_t  dataLen;       // 이 조각의 실제 길이
  uint8_t  commentDay;    // 멘트 생성 날짜
  char     data[200];     // 조각 데이터
} AIChunk;

A보드는 최대 3회 재시도하며, 응답이 없으면 RTC 메모리의 캐시 데이터를 사용합니다.

05B 보드 — 게이트웨이

WiFi & AP 모드

저장된 자격증명으로 접속하고, 실패 시 DualCore-Setup AP를 띄워 웹 설정을 받습니다.

외부 API

Open-Meteo(날씨), Air Quality(대기질), Gemini(생활 조언 1줄)를 호출합니다.

NTP 동기화

한국 시간(KST, GMT+9)으로 시각을 맞추고 A보드에 함께 전달합니다.

웹 설정 페이지

위치·기상시각·타임아웃·AI on/off 변경 및 공장 초기화를 제공합니다.

AI 멘트 생성 (Gemini)

오늘의 기상·대기질을 프롬프트로 넣어 영어 18단어 이내 생활 팁을 하루 1회 생성하고 NVS에 저장해 재부팅 후에도 유지합니다.

06A 보드 — e-paper 디스플레이

화면 모드

Calendar(일/월 보기) · Weather(6시간대 예보) · Air/Env(종합+AI 멘트). UP/DOWN으로 모드 전환, OK로 새로고침, EXIT로 시스템 Info 진입.

렌더링 & 전력 관리

  • 부분 갱신으로 빠르게 그리되, fullRefreshEvery마다 전체 갱신으로 잔상 제거.
  • 날씨 코드별 24×24 아이콘(맑음/구름/비/눈/안개/뇌우)과 AI 멘트 워드랩 표시.
  • 입력이 없으면 캘린더 복귀 후 Deep Sleep, wakeHour까지 타이머로 기상.

다음 기상 시각(wakeHour) 계산

NTP로 시간이 동기화된 상태라면, 현재 시각(hour:minute)에서 설정된 기상 시각(wakeHour)까지 남은 초를 계산해 타이머로 잠듭니다. 아직 오늘 기상 시각이 지나지 않았으면 오늘 기준, 이미 지났으면 다음 날 기준으로 계산합니다.

if (hour < wakeHour)            // 오늘 아직 기상 시각 전
  sec = (wakeHour - hour) * 3600 - minute * 60;
else                            // 이미 지남 → 다음 날 기상
  sec = (24 - hour + wakeHour) * 3600 - minute * 60;

if (sec <= 0) sec = 3600;       // 안전장치: 최소 1시간

예) wakeHour=7, 현재 22:30 → 다음 날 07:00까지 약 8시간 30분 후 기상. 시간이 한 번도 동기화되지 않았으면 기본 24시간 주기로 기상합니다.

RTC 메모리 활용: RTC_DATA_ATTR 변수로 시간·날씨·AI 멘트·설정을 Deep Sleep 동안 유지해, 통신 실패 시에도 마지막 데이터를 표시합니다.

07설정 동기화 메커니즘

B보드에서 바꾼 설정을 A보드에 안전하게 전파하기 위해 버전 번호(configVersion) 방식을 사용합니다.

  • 웹에서 설정 저장 시 B보드의 configVersion이 1 증가.
  • 설정값은 매 센서 응답에 실려 A보드로 전달.
  • A보드는 받은 버전이 현재 적용 버전보다 클 때만 적용.
  • 같은 설정이 반복 수신돼도 불필요한 갱신 없이 변경 시점만 정확히 반영.
// A보드 수신 콜백
if (rxMsg.config.configVersion > lastCfgVersion) {
  appliedCfg     = rxMsg.config;          // 새 설정 적용
  lastCfgVersion = rxMsg.config.configVersion;
}

08동작 시나리오

  1. B보드(상시 동작): WiFi로 날씨·대기질을 가져와 센서값과 함께 저장하고, Gemini에 AI 멘트를 요청해 NVS에 보관합니다. A보드의 요청을 항상 대기합니다.
  2. A보드 기상 & 데이터 수신: 타이머 또는 버튼으로 Deep Sleep에서 깨어나 ESP-NOW로 B보드에 요청해 센서·시간·날씨·대기질·설정을 받고, Air/Env 모드면 AI 멘트도 수신합니다.
  3. 통신 실패 시 데이터 처리: 재시도 후에도 응답이 없으면 각 값은 다음과 같이 불러옵니다.
    • 시간·온습도·날씨·대기질 → RTC 메모리(RTC_DATA_ATTR)에 남은 마지막 수신값(캐시) 사용.
    • AI 멘트 → 같은 날짜의 캐시가 있으면 그대로 표시, 없으면 "Requesting AI..." 안내 표시.
    • 설정값 → 직전에 적용된 appliedCfg(RTC 유지)를 그대로 사용.
  4. 화면 렌더링: 현재 모드(Calendar / Weather / Air-Env / Info)에 맞는 화면을 e-paper에 그립니다.
  5. 타임아웃 & 복귀: 웹서버에서 설정된 타임아웃(sleepTimeoutSec) 동안 입력이 없으면 어떤 화면에 있든 캘린더 화면으로 복귀 후 Deep Sleep에 재진입합니다.
  6. 슬립 표시: Deep Sleep 진입 직전, 화면 왼쪽 위에 작은 점(인디케이터)을 부분 갱신으로 그려 현재 잠든 상태임을 표시합니다.

09빌드 & 설정

  • 라이브러리: WiFi, esp_now, WebServer, Preferences, Adafruit_SSD1306, DHT, ArduinoJson, GxEPD2
  • MAC 매칭: peerMAC_A / peerMAC_B를 상대 보드 실제 MAC으로 설정
  • ESP-NOW 채널 자동 탐색: 고정 채널이 아니라 A보드가 savedChannel을 우선 시도하고, 실패 시 1~13 채널을 스캔해 B를 찾음. 성공한 채널은 RTC에 저장해 다음 부팅 때 우선 사용 (스캔 실패 시 폴백값은 DEFAULT_CHANNEL = 10)
  • Gemini API 키: GEMINI_API_KEY에 본인 키 입력 (소스 노출 주의)

10한계 & 개선점

  • API 키 하드코딩: 소스 노출 → NVS 저장 등으로 분리 권장.
  • AI 멘트 갱신: 하루 1회 — 날씨 급변 반영이 늦을 수 있음.
  • ESP-NOW 비암호화: 현재 encrypt = false, 필요 시 암호화 적용 가능.
  • 고정 위치: 좌표는 변경 가능하나 AI 프롬프트 도시명은 고정.
